# Connected Retro - 2026-07-12

HTB Medium Linux machine. FreePBX 16.0.40.7 with pre-auth SQLi to RCE, incron privesc
to root. Both flags captured across two box instances and ~3 hours of wall time.

## What happened

The initial foothold (CVE-2025-57819, pre-auth SQLi chained to file upload RCE) worked
on the first try. The root privesc (incron `ha_trigger` -> malicious FreePBX HA module
-> SUID bash) was identified within 20 minutes but took over 2 hours to execute because
**incron watches don't fire from Apache/webshell context**. Every trigger method we tried
from the webshell failed silently. The fix was planting an SSH key and triggering from a
real SSH session, where inotify events propagate correctly.

## Attack chain (what worked)

1. **CVE-2025-57819: Pre-auth stacked SQLi.** The Endpoint Manager's `brand` parameter
   in `ajax.php` accepts unescaped input into a SQL query. Stacked queries create a new
   admin user in the `ampusers` table. Public PoC (0xEhab) worked against 16.0.40.7 with
   `--http` flag.

2. **CVE-2025-61678: Authenticated file upload.** The Endpoint Manager firmware uploader
   has a path traversal in the `fwbrand` parameter. Traversing out of `/tftpboot/customfw/`
   into `/var/www/html/` drops a PHP webshell. The PoC chains both CVEs automatically.

3. **SSH key plant.** Created `~asterisk/.ssh/authorized_keys` via the webshell. This gave
   a proper login shell that bypasses whatever namespace/cgroup restriction blocked inotify
   events from the httpd worker processes.

4. **incron ha_trigger privesc.** Created `/var/www/html/admin/modules/freepbx_ha/` (doesn't
   exist by default) with a malicious `functions.inc/incron.php` containing a class with
   `rootTrigger()` that copies `/bin/bash` to `/tmp/rootbash` and sets SUID. The
   `/usr/sbin/sysadmin_ha` script (run as root by incron on `IN_CLOSE_WRITE` to
   `ha_trigger`) includes this file and calls the method. `touch /usr/local/asterisk/ha_trigger`
   from SSH triggers it. `/tmp/rootbash -p` gives root.

## What broke (tooling gaps)

### 1. Wasted 2+ hours on a context-dependent inotify issue

**What happened:** The incron privesc was set up correctly within 20 minutes of landing
the webshell. We then spent over 2 hours trying every trigger method imaginable: PHP
`file_put_contents`, bash `echo >`, `dd`, compiled C program with explicit `open/write/close`
syscalls, `perl` open/close, `nohup` backgrounded scripts, `at` scheduled jobs. Nothing
fired the incron watch.

The fix was trivial: `touch ha_trigger` from an SSH session.

**Why:** The Apache httpd workers run in a systemd cgroup (`httpd.service`). The webshell
inherits this context. While the raw inotify syscalls work from this context (we verified
with Python ctypes), the incrond process apparently doesn't observe `IN_CLOSE_WRITE` events
generated by processes in the httpd cgroup. This could be due to mount namespace isolation,
audit subsystem interaction (laurel was running), or a quirk of the SNG-patched incron
(incron-0.5.10-11.sng7).

**What PK needs:**
- The exploitation skill should document: "If incron/inotify-based triggers don't fire from
  a webshell, try from a different process context (SSH, cron, `at` job). Plant an SSH key
  and trigger from there."
- More generally: when a privesc technique is confirmed correct but doesn't fire, **try it
  from a different execution context** before assuming it's broken.

### 2. Massive context burn on dead-end enumeration

**What happened:** Before confirming incron as the path, we investigated 10+ privesc
vectors in series: PwnKit (patched), Baron Samedit (patched), DirtyPipe (kernel too old),
MySQL FILE priv (denied), Redis config write (wrong user), MongoDB (no client), modprobe
install directives (kernel doesn't trigger them on auto-load), CVE-2023-32233 nf_tables
(exploit fails in user namespace), fwconsole replacement (only runs as root at midnight),
and sudo password brute force.

The orchestrator context filled with raw enumeration output from all these attempts, making
later reasoning slower and more expensive.

**What PK needs:**
- Delegate privesc enumeration to a single agent with a structured checklist, not inline
  in the orchestrator context. The agent reports back a short summary, not raw output.
- The orchestrator should not run more than 3-4 `pk exec` commands directly before
  spawning an agent. This session ran 50+ commands inline.

### 3. No pre-built SSH key relay

**What happened:** When we needed to switch from webshell to SSH, we had to: generate a
key pair in the attackbox, extract the public key, install it via the webshell, then SSH
in. This took 3 commands and worked, but only because we thought to try it.

**What PK needs:**
- `pk ssh plant <webshell_url>` or similar that auto-generates a key, installs it via the
  webshell, and verifies SSH access. This is a common pivot pattern.

### 4. Exploit agent ran for 70 minutes on dead ends

**What happened:** The exploit agent (spawned for root privesc) ran for ~70 minutes and
305 tool calls. It found several useful things (asterisk in sudoers, modprobe.d writable,
nf_tables vulnerability) but couldn't get root. It spent significant time on
CVE-2023-32233 exploitation (cross-compiling, transferring, debugging) which ultimately
failed.

**Why:** The agent brief didn't prioritize "try the known intended path from a different
context" because we didn't know that was the issue yet. The agent correctly explored
alternative vectors but each one was expensive.

**Fix:** Agent briefs for privesc should include: "If the primary vector is set up but
doesn't trigger, try executing the trigger from a different process context (SSH, cron, at)
before moving to alternative vectors."

## What worked well

- **CVE lookup and PoC-first approach.** The fork agent found CVE-2025-57819 with public
  PoC in 60 seconds. The PoC worked on first run. Total time from "FreePBX 16.0.40.7" to
  "uid=999(asterisk)": under 3 minutes.

- **Engagement setup discipline.** Created the engagement, added targets, logged findings
  and activity throughout. The DB has a complete audit trail.

- **Identifying the incron path quickly.** Reading the sysadmin_ha script, noticing the
  freepbx_ha module directory doesn't exist, and realizing we can create it with a malicious
  PHP class: that whole analysis took about 15 minutes.
