# Retro: Jump CTF (THM) — 2026-06-30

THM room "Jump" — lateral movement chain across 5 users on a single Linux host.
First real playbook-driven engagement after the graph runtime shipped.

**Result:** 5/5 flags captured, 79/79 playbook steps resolved (40 done, 39 skipped).

## Kill chain

```
anonymous FTP → recon_user (cron exec)
             → dev_user   (group membership)
             → monitor_user (PATH hijack via grep in healthcheck.service)
             → ops_user   (deploy_helper.sh overwrite + sudo)
             → root       (sudo less + shell escape)
```

## What went well

1. **Playbook auto-skip worked.** Web recon block (6 steps), SSL, SMB, SNMP all correctly
   skipped when no matching services were found. Saved time and agent tokens.

2. **Recon agent was fast and surgical.** 43 tool calls, 205s. Found both ports, enumerated
   FTP structure, identified the automation pipeline. Clean handoff to enum.

3. **`grep` hijack over `ps` hijack.** The second exploit agent figured out that creating a
   new binary (`/opt/dev/bin/grep`) bypasses bash's hash table, while replacing an existing
   one (`ps`) doesn't. This is a real technique insight.

4. **Mapped chain survived machine reset.** When the THM room timed out, the full chain was
   documented in the engagement DB. The final agent replayed all 5 flags in one pass (92
   tool calls, 1313s) because every pivot was already mapped.

5. **Objective tracking.** Flag capture via `pk objective capture` kept score in the DB.
   Quick status checks throughout the engagement worked well.

## What went wrong

### 1. First exploit agent stalled on a failing technique

**What happened:** The first exploit agent spent 438 tool calls (~192k tokens, ~45 minutes)
retrying the same `ps` PATH hijack. Bash's `hashall` option caches resolved command paths
in the running shell's hash table. The healthcheck process had been running since boot, so
`ps` was already hashed to `/usr/bin/ps`. The agent kept uploading new payloads to
`/opt/dev/bin/ps` and waiting for execution that never came.

**Why:** No stall detection. The agent had no instruction to recognize when it's looping on
the same failed approach. It also didn't consider alternative binaries or alternative pivot
paths.

**Fix:**
- Agent briefs should include: "If a technique fails twice with the same symptoms, stop and
  try a different vector. Report back if you exhaust 3 distinct approaches."
- The exploitation skill should document the bash hash table issue under PATH hijack
  techniques: "Prefer creating a NEW binary name that the target script calls but hasn't
  executed yet. Replacing an existing binary only works if the process restarts or runs
  `hash -r`."
- Consider a token/call budget per agent. 200 tool calls without a new finding should
  trigger an automatic report-back.

### 2. Agents ignored `pk exec` discipline

**What happened:** Both exploit agents used raw `curl`, `bash`, and other commands directly
instead of `pk exec -- <command>`. The activity log has gaps covering the entire exploit and
post-exploit phases. Only recon-phase activity was logged.

**Why:** The agent briefs said "use `pk exec`" but didn't explain why or make it a hard
requirement. Agents optimized for speed and skipped the wrapper.

**Fix:**
- Agent brief language should be stronger: "Every command that touches the target MUST use
  `pk exec`. This is the audit trail. Raw bash/curl calls are invisible to the engagement
  log and will not be accepted."
- Consider: can we enforce this at the hook level? A PreToolUse hook that warns when Bash
  is called with target-facing commands without `pk exec` prefix.
- The exploitation skill procedure should list `pk exec` in step 6 alongside
  `pk activity log`, not as a separate afterthought.

### 3. Playbook has no lateral movement loop

**What happened:** The privesc block assumes a single hop: user → root. This room had 4
lateral movements (recon → dev → monitor → ops → root), each requiring its own
enum-exploit mini-cycle. We handled it through freestyle steps and explicit agent
instructions, but the playbook structure didn't help.

**Why:** The CTF playbook was designed for the common case (initial access + one privesc).
Multi-user lateral movement is common in THM/HTB rooms and real engagements.

**Fix:**
- Add a "lateral movement" block to the playbook that repeats per pivot:
  - Enumerate current user context (id, groups, sudo -l, cron, writable files, SSH keys)
  - Identify trust boundary to next user
  - Exploit the boundary
  - Capture flag/evidence
  - Repeat until target user reached
- The block should be dynamically instantiated — the orchestrator shouldn't need to know
  the user count upfront. Could be a "repeat until root" loop or explicitly instantiated
  per discovered user.
- This is a playbook template change, not a runtime change. The BT graph already supports
  dynamic step insertion via `pk step add` (if we build it) or could be handled as a
  freestyle block with structured sub-steps.

### 4. Target timeout mid-exploitation

**What happened:** The THM room timed out while the exploit agent was working on the
monitor → ops pivot. The agent continued polling for ~80 minutes (220 tool calls, ~5000s)
before giving up, burning tokens on a dead target.

**Why:** No connectivity health check. The agent didn't detect that the target was
unreachable and kept uploading scripts and waiting for results.

**Fix:**
- Agent briefs should include: "Before each exploit attempt, verify the target is reachable
  (`pk exec -- nmap -Pn -sT -p 21 <ip>` or similar). If the target is down, report back
  immediately. Do not retry more than twice."
- Could also be a runtime check: if 3 consecutive `pk exec` commands time out or return
  connection refused, auto-abort and notify the orchestrator.

### 5. Root privesc needed a nudge

**What happened:** The final exploit agent found that ops_user has
`sudo NOPASSWD: /usr/bin/less` with `env_keep+=LESS`, but couldn't execute the shell
escape autonomously. I had to send a message with the specific GTFObins technique
(`!` command in less, LESSOPEN pipe).

**Why:** The agent knew the vector but couldn't figure out how to exploit `less` non-
interactively (it needs stdin interaction to type `!/bin/sh`). It tried LESSOPEN and LESS
env vars but those were blocked by sudo's env_reset (only LESS was kept, not LESSOPEN).
The working technique was using python3 subprocess to pipe `!` commands to less's stdin.

**Fix:**
- The exploitation skill should include a "common sudo privesc patterns" reference,
  especially for interactive programs (less, vim, man, more, nmap --interactive). For
  non-interactive exploitation, document the python3 subprocess pipe technique.
- Consider shipping a GTFObins-style lookup as a playbook block or reference that agents
  can consult when they find a sudo entry.
